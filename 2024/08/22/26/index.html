自定义调度器方法

一般来说，我们有 4 种扩展 Kubernetes 调度器的方法。

  • default-scheduler recoding: 直接在Kubernetes默认scheduler基础上进行添加，然后重新编译kube-scheduler
  • standalone: 实现一个与kube-scheduler平行的custom scheduler，单独或者和默认kube-scheduler一起运行在集群中，使用.spec.scheduler.name来区分使用哪一个调度器。
  • scheduler extender: 实现一个”scheduler extender”，kube-scheduler会调用它(http/https)作为默认调度算法(预选&优选&bind)的补充，只支持在 Filter、Prioritize、Preempt 和 Bind 的扩展
  • scheduler framework: 实现scheduler framework plugins，重新编译kube-scheduler，类似于第一种方案，但是更加标准化，插件化

default-scheduler recoding 以及 standalone方案都属于侵入式的方案，不太优雅；而 scheduler extender 以及 scheduler framework 属于非侵入式的方案，这里重点介绍 scheduler extender，开发自己的调度器插件，代码基于 kubernetes 的v1.28.2版本。

Extender工作原理

scheduler extender 类似于 webhook，kube-scheduler会在默认调度算法执行完成后以 http/https 的方式调用 extender，extender server 完成自定义的预选&优选逻辑，并返回规定字段给scheduler，scheduler结合这些信息进行最终的调度裁决，从而完成基于extender 实现扩展的逻辑

配置 Extender

在使用 extender 之前，我们必须在 KubeSchedulerConfiguration 配置文件中定义 extender 相关信息，如下：

pkg/scheduler/apis/config/types.go#267

这里面主要关注如下几个字段：

  • URLPrefix：extender 访问地址
  • FilterVerb：表示过滤节点的 API 路径
  • PreemptVerb：表示抢占的 API 路径
  • PrioritizeVerb：表示打分节点的 API 路径
  • Weight：表示 extender 优选算法对应的权重
  • BindVerb：表示 bind 节点的 API 路径
  • NodeCacheCapable：如果 Extender 支持缓存节点信息，设置为 true，默认调度器只会发送符合 node 节点的基本信息
  • ManagedResources：定义由 Extender 管理的资源。如果某些资源的调度不应由默认调度器处理，设置 ignoredByScheduler 为 true
  • Ignorable: 如果设置为 true，调度器在 Extender 不可用时不会失败。

profiles：可以配置默认调度器，以及基于 scheduler-framework框架自己开发的调度插件

extenders：配置 extender 的相关信息
